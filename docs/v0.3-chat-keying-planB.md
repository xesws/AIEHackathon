# v0.3 — CHAT 路径编辑触发修复:query-span isolation (Plan B)

## 背景与问题

SPIKE 0 (v0.2) 证明 RAW-prompt 路径绿:`edit → retrieve → answer` 成立。
但 **CHAT 推理路径失败** —— 编辑后用 chat 模板提问,编辑不触发(模型答
"I'm not aware of any information about ...")。

**根因(已确认)。** HoReN 的检索 key 是输入前向在 layer-29 的隐状态。
`HopfieldAdapter` 替换了 `model.layers[29].mlp.down_proj`,key =
`_select_query(args[0], key_id)`(策略 `last_60_perc_prompt_tokens_avg`)→ L2 归一化 →
`_query`(归一化 Hopfield ≈ cosine)打分 → 阈值 `0.85` 门控。
- WRITE(`HOREN.edit`):前向跑在 RAW `"{stem} {target}"`,key 落在 raw-stem token 上。
- READ(`HOREN.generate`):`key_id = input_ids.shape[1]-1`,key 落在 **chat 模板化**序列上
  (`<bos> system + RAG窗口 + user-turn + assistant header`)。

于是 read-key 把固定的 scaffold(system + RAG窗口 + role headers)平均进了 key,偏离 raw
write-key,打分 < 0.85,编辑不触发。

## 修复 = Plan B:在**隐状态张量**层面拆 query(不是拆文本)

整条 templated prompt 跑一次 forward,在 layer-29 拿到 `[seq_len, hidden_dim]`;做 key 时
**只取 user-turn(query)token 对应的那几行**,排除 `[sys] [RAG窗口] [role headers]` 行。
write 与 read 用**完全相同**的切片规则。

```
  layer-29 隐状态(整条 templated prompt):
  ┌───────────────────────────────────────────────────────────────┐
  │ [sys 行] [RAG窗口 行] [role-header 行] [QUERY-span 行]            │
  └───────────────────────────────────────────────────────────────┘
                                          └─ key = mean(这几行) ─┘
  Plan A(放弃):整条 / 末 token / 末60%平均 → 前缀污染 key,且不同 fact 的 key 坍缩 → locality 崩
  Plan B(本方案):只切 QUERY-span 行 → 前缀基本不进 key
```

Plan B 同时解决 ①(write/read 同分布)**和** locality 坍缩,且对非空 RAG 窗口(未来)天然鲁棒。
**值注入位置不变**(仍是末 token = 现 `HOREN.generate` 的 `key_id`),保住值迁移几何;只改 key 计算。
② (paraphrase 容忍)本轮**只测不修**。

## 关键决策

1. **写 key = APPEND 不 overwrite。** `editing.edit` 保留 HoReN 原生 raw key,**追加**一个 Plan-B
   chat key,复用同一个已训练的 value。一次编辑同时服务 raw 路径(命中 raw key)和 chat 路径
   (命中 chat key)→ SPIKE 0 全绿 + chat 修复。这是同一 stem 的两种 render,不是 paraphrase
   multi-key,不碰 ②。由 `key_mode` 开关:`'chat'`(默认)追加 chat key;`'raw'` = legacy。
2. **注入位置不变**(末 prompt token)。只改 key 计算。
3. **chat key pooling = query span 整段 mean**;raw 模式保留 HoReN 原 `_select_query`(让 grid 的
   raw 格反映真实行为)。
4. **范围 = hero**(固定 system + 空 RAG 窗口)。机制已可推广到非空窗口,但本轮只接线/验证空窗口。

## 改动文件

- `third_party/horen/src/models/horen/editor.py`(唯一 vendored 改动,最小):
  `HopfieldAdapter` 加 `self.query_span=None`、加 `_pool_span(layer_input,start,end)` 辅助
  (forward 与 compute_key 共用,提取逻辑不分叉);`forward` 里当 `query_span` 非空时用
  `_pool_span` 取 key,否则走 legacy `_select_query`;注入仍在 `last_prompt_token_index`。
  `HOREN.generate`/`HOREN.edit`/阈值/`_query`/key 向量空间均不动。
- `keying.py`(新,repo 根 = adapter 胶水):`locate_query_span`(用 `apply_chat_template` +
  `return_offsets_mapping` 定位 query token 行,排除 special/role token)、`compute_key`
  (forward_pre_hook 抓 `args[0]` + `adapter_mode='none'` 单次前向,复用 `_pool_span`/`_select_query`)、
  `score`(临时塞 codebook 用 `adapter._query` 打分,即生产门控函数)。
- `editing.py`:`edit(..., key_mode='chat')`,编辑后追加 Plan-B chat key(复用 value 行)。
- `generate.py`:chat 分支设 `adapter.query_span = locate_query_span(...)`,raw 分支设 `None`
  (防跨调用污染);其余不动。
- `spikes/spike0_planB_keying_grid.py`(新):secondary 测量脚本(诊断 grid + locality 探针)。

## 测量(secondary,只测不修)

取 `eval/samples.json` 前 5 个 `sample_type=="A"`(stem=`edit_prompt`,
paraphrase=`queries[0].q`,target=`target_new`):

(a) **诊断 grid** —— codebook = 单个 RAW key `compute_key(stem,raw)`,对每个 read-key 打分:

```
                    read raw          read chat (Plan B)
  read = stem       [s]/PASS  ~1.0    [s]/PASS   ① 单独(已 span 隔离)
  read = paraphrase [s]/PASS  ②裁决   [s]/PASS   ①+②
  read = UNRELATED   —                [s]        locality 探针(应 LOW)
```
裁决:raw/stem = 对照 ~1.0;chat/stem = ① 成本;raw/paraphrase = ② 裁决(≥0.85 → ② 非问题;
<0.85 → ② 真,留 C/D);chat/paraphrase = ①+②。

(b) **修复验证** —— codebook = 单个 CHAT Plan-B key `compute_key(stem,chat)`,读 chat/paraphrase
(应 ≈ raw/paraphrase 格 → ② 保住、① 去除)+ 读 chat/UNRELATED(应 LOW → **Plan B 修好 locality**)。

## e2e 验证(hero loop:chat 路径,RAG OFF,fresh session,2–3 个 type-A)

新 `load_base`;`editing.edit(..., key_mode='chat')` 编一个 fact;空 RAG 窗口;走 chat 路径:
1. **stem 提问** → 触发(score ≥0.85)且生成含 `target_new`。← ① 干净证明,把 SPIKE 0 step[6] 转绿。
2. **paraphrase 提问** → 报 score + 是否含 target(① + ② 信息;若 ② 真可能 miss,属预期)。
3. **unrelated 提问** → 不触发(score < 0.85)→ locality 完好。
另:重跑 SPIKE 0 确认 raw 步 [2]-[5],[7] 仍绿(append 保住 raw key)。
