## 为什么这么做(两条机制事实,已读码 verify)
1. **归因机制上是「每条回答一次检索决策」,不是逐 token。** live 解码 `use_cache=False` + `HOREN.generate` 把 `key_id`
   锁在 prompt 最后一个 token(`third_party/horen/.../editor.py`),所以每个生成 token 共享**同一个**
   `(命中槽 chosen_key, 相似度, 是否过 0.85 阈值)`。mock 的逐 token 彩虹机制上拿不到 → **诚实做成每条回答级**
   (不改解码、不追假逐 token)。
2. **codebook 槽 → MemoryItem 的映射当前不存在**(`PROV_EDIT_REF=id(adapter)` 对所有条目相同)。固化时
   `editing.edit` 返回的 `codebook_size` / `wrapper.edit_log["chosen_key"]` 被丢弃 → 本轮把 key index 记进 `provenance`
   (additive,数据现成)。
3. **搜索后端现成**:`memory/rag_store.search(q,k)->list[MemoryItem]`(embedding 召回 + LLM 重排,语义),`/chat` 已在用。
