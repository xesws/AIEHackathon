> 本轮目标(一句话):under-the-hood 面板里的 **edit module 开关** 在「没有写入权重(codebook k=0)」时
> 仍可点 → 反复打出 409(**F3**);开关文案把「写入(consolidate)」和「挂载/热插拔(on/off)」两个概念
> 混在一起(**F4**);并复核「consolidate 安装 adapter 后开关是否与 server 真值 desync」(**C3**)。
> 只动 `frontend/src/engram.jsx`(+ rebuild)与必要的 serving **edit-module 路由测试**;**不碰** chat/attribution/
> 检索/codebook/β/阈值。

## 一、计划节

### 1. 根因(读代码确认)

#### F3 —— k=0 时开关仍可点,反复 409
- `engram.jsx:703` `toggleEdit(next)` 是**乐观更新**:先 `setEditOn(next)`,再 `await apiEditModule(next)`,
  失败才回滚 `setEditOn(!next)` + `setBackendErr(true)`。
- server 端 `serving/app.py:209` `edit_module()`:`on==true` 且 `model_host.recorded_adapter() is None` →
  **抛 409**「no edit module to enable」。`recorded_adapter()` 只有在**第一次成功 consolidate**(`editing.edit`
  → `register_edit_module`,见 C3)之后才非空。
- ★ 因此「**codebook k=0**(`codebookK = weights.length`,`weights` ← `/memories.consolidated`)」⟺「没有 recorded
  adapter」⟺「点 ON 必 409」。当前开关在 k=0 时**不禁用** → 用户反复点 → 反复 409(乐观更新闪一下又弹回),
  且 backendErr strip 是泛化文案「刚才那步没成功…后端可能在忙」,**对 409 是误导**(根本不是后端忙,是还没写权重)。
- 注:raw 字符串「409」其实**没有**直接渲染给用户(`catch (e)` 只 `setBackendErr(true)`,不渲染 `e.message`)
  —— 但反复弹泛化错误条同样糟。

**修法(最小,前端)**:`codebookK === 0` 时**禁用** edit-module 的 `Switch`,并把 hint 换成人话
「`还没有写入权重 — 先 consolidate`」。禁用后该 409 路径从 UI **不可达** → 既不弹泛化错误条、也杜绝任何 raw 409。
- `Switch` 组件加可选 `disabled` prop(`onClick` 时 `!disabled` 才 `onChange`;`<button disabled>`;视觉降透明 +
  `cursor:not-allowed`)。**additive**:RAG 开关不传 `disabled` → 行为字节不变。
- `codebookK` 在 `LabPanel` 已是 prop(`engram.jsx:203` + 调用处 `codebookK={weights.length}`),直接用。

#### F4 —— 开关文案混淆「写入」与「挂载」
- `engram.jsx:236` 现文案「`权重里的记忆 · 关掉即拔出(热插拔)`」:「权重里的记忆」像是在说「写入这个动作」,
  而开关其实只做**运行时挂载/拔出(hot-swap)**,**不**做持久化写入(写入 = consolidate)。两个概念混在一起,
  观众会以为「关掉开关 = 没写进权重」。
- 仓库别处已用 mount/hot-swap 语言:`engram.jsx:289`「detached — adapter 已拔出」、`:424`「这些记忆当前未挂载到模型」
  —— 只有这条 kill-switch hint 用词不一致。

**修法(只改文案,前端)**:k>0 时 hint 改为「`挂载/拔出已写入的 adapter(热插拔)· 写入交给 consolidate`」,
明确「这个开关 = 挂载/生效(on/off),持久化写入是另一个 consolidate 动作」。**不重排布局**,只换 hint 文本。

#### C3 —— consolidate 安装 adapter 后是否与开关 desync(**需确认**)
代码路径:
- `serving/app.py:240` lifespan 注入 `consolidate.set_model_provider(lambda: model_host.current_model())`。
- consolidate(`memory/consolidate.py:142` `_process_item`)对每条 edit-route item 调 `editing.edit(model, req)`。
- `editing.py:62` `apply_horen_to_model(...)` 把 `HopfieldAdapter` **装到 inner_params slot**;`:65` `adapter =
  model_host.edit_module()`(取回刚装上的 adapter);`:69` `model_host.register_edit_module(adapter, wrapper)`
  → `model_host.py:109` 把 `_S["adapter"]=adapter` 并把 resident `_S["model"]` 换成 HOREN wrapper。
- 于是 consolidate 后 `model_host.edit_active()`(`model_host.py:98`:`_S["adapter"] is not None and
  getattr(parent, attr) is _S["adapter"]`)= **True** —— **即使用户之前把开关切到 OFF**。
- `/health`(`app.py:230`)`edit_on = model_host.edit_active()`(slot 真值)。
- 前端 `refresh()`(`engram.jsx:622`)每次都 `const h = await apiHealth(); setEditOn(h.edit_on)`;且
  consolidate 路径(`consolidate()` / `burnOne()`)在 `apiConsolidate*()` 之后**必调** `refresh()`。

**待确认的判断**:开关是否会与 server 真值留下 **残留 desync**?

**初判(将在结果节给最终结论 + 证据):** 倾向 **不是真 desync bug**。理由链:
1. `/health.edit_on` = `edit_active()` = **slot 真值**(`model_host.py:98-103`,无任何缓存/影子状态)。
2. 前端开关在每次 `refresh()` 都**镜像** `/health.edit_on`,而 consolidate 后**一定**走 `refresh()`。
3. ⇒ consolidate 后开关 = ON,server 也 = ON,**两者一致**(开关诚实跟随,不存在「开关显示 X、server 是 Y」的残留偏差)。
4. 「consolidate 把 OFF 顶成 ON」是**真实行为**,但属 **by-design**:`editing.edit` 必须把 adapter 装上 slot 才能让
   这次编辑生效;且 demo hero-loop(见 `docs/v2.4` G5)**明确期望** consolidate 后 `edit_on=true`。开关把这点**如实**反映
   出来(翻到 ON),用户可再手动 OFF。
   - 故 prompt 给的「**优先**修法 = 确保 `/health.edit_on` 反映 slot 真值且前端开关始终镜像它」**本就已满足**;
     另一选项「让 consolidate 恢复用户 OFF 意图」会**反过来打破** hero-loop 的 `edit_on=true` 期望(回归风险),不取。

→ 结论方向:**C3 证伪(非 desync bug),不改 production 代码**;仅在 `tests/test_serving_routes.py` 补一条
**纯 CPU lock 测**,锁住「`/health.edit_on` 双向镜像 `edit_active()`」这条不变量(现有测试只覆盖 False 半边,补 True 半边)。
最终确认/证伪以结果节为准。

## 二、结果节

> **结论:F3/F4 前端最小修复落地;C3 经代码路径确认为「非 desync bug」(证伪),production 代码不动,仅补一条
> 纯 CPU lock 测锁住不变量。** `pytest` 三个目标文件 **43 passed**(含新增 C3 lock);前端已 `build.py` rebuild,
> `index.html` 与 source 同步。⚠ F3/F4 的浏览器帧验证(开关真禁用 + 新文案显示)需手测一帧,本轮 headless 做不了。

### 1. F3 —— k=0 禁用 edit-module 开关(杜绝反复 409)
- `Switch` 组件加 **additive** `disabled` prop:`onClick` 里 `!disabled && onChange(!on)`、`<button disabled>`、
  视觉 `opacity:0.4` + `cursor:not-allowed`。RAG 开关不传 → 行为字节不变。
- edit-module `Switch` 传 `disabled={codebookK === 0}`(`engram.jsx:248`)。compiled bundle 实证:
  `{on:t,onChange:l,label:"edit module",disabled:n===0}`(`n` = minified 后的 `codebookK`)。
- k=0 时该 409 路径从 UI **不可达** → 不再弹泛化错误条、也不会有任何 raw 「409」到达用户。

### 2. F4 —— 开关文案区分「挂载/热插拔」与「写入/consolidate」
新引出的两段 hint(`engram.jsx:243-246`,按 `codebookK` 二选一):

| 条件 | 旧文案 | 新文案 |
|---|---|---|
| k=0(F3) | (无,且开关可点) | **`还没有写入权重 — 先 consolidate`** |
| k>0(F4) | `权重里的记忆 · 关掉即拔出(热插拔)` | **`挂载/拔出已写入的 adapter(热插拔)· 写入交给 consolidate`** |

新 k>0 文案把「这个开关 = **挂载/生效(hot-swap on/off)**」与「**写入/持久化 = consolidate**」明确分开,
与仓库别处用语(`:289` 「detached — adapter 已拔出」、`:424`「未挂载到模型」)一致,观众不再误以为「关开关 = 没写进权重」。

### 3. C3 —— **证伪(非 desync bug),不改 production 代码**
**确认结论:开关与 server 真值之间无残留 desync。** 代码路径证据:
1. `/health.edit_on`(`serving/app.py:230`)= `model_host.edit_active()`(`model_host.py:98-103`)= **slot 原始真值**
   (`_S["adapter"] is not None and getattr(parent, attr) is _S["adapter"]`),**无任何影子/缓存状态**。
2. 前端开关在**每次** `refresh()`(`engram.jsx:622` `setEditOn(h.edit_on)`)镜像 `/health.edit_on`;而 consolidate
   两条路径 `consolidate()`(`:650`)/ `burnOne()`(`:664`)在写入后**必调** `refresh()`。
3. consolidate 经 `editing.edit`(`editing.py:62/65/69` → `register_edit_module`,`model_host.py:109`)把 adapter
   装上 slot ⇒ `edit_active()` 翻 True ⇒ refresh 后开关=ON、server 也=ON,**两者一致**。不存在「开关显示 X / server 是 Y」的残留偏差。
4. 「consolidate 把用户先前的 OFF 顶成 ON」是**真实**但 **by-design** 的行为:`editing.edit` 必须装 adapter 这次编辑才生效;
   且 demo hero-loop(`docs/v2.4` G5)**明确期望** consolidate 后 `edit_on=true`。开关**如实**反映该状态(翻 ON),用户可再手动 OFF。
   - prompt 给的「**优先**修法 = `/health.edit_on` 反映 slot 真值且前端始终镜像它」**本就满足**;另一选项「让 consolidate
     恢复用户 OFF 意图」会**反过来打破** hero-loop 的 `edit_on=true` 期望(回归风险),故**不取**。

**C3 不改 production 代码。** 仅补一条纯 CPU lock 测 `test_health_edit_on_mirrors_edit_active_true`
(`tests/test_serving_routes.py`):现有 `test_health_reports_ready_edit_on_and_counts` 只覆盖 `edit_active()=False` 半边,
新测补 `edit_active()=True → /health.edit_on=True` 半边,锁住「`/health.edit_on` 双向镜像 `edit_active()`」这条让 desync
不可能发生的不变量(mock 掉 model_host,不碰 GPU)。
