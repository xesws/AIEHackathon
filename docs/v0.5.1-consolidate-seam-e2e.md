> 阶段:集成纵切 / 串行 / GPU 常驻。开发前写计划,开发后补结果(遵 CLAUDE.md /docs 规则)。

> spike 里验过 —— `consolidate.run_pass → 真 editing.edit → 真 chat 检索` **这一跑从未发生**。

## 目标
用一个 driver,把 **buffer → `consolidate.run_pass` → 真 HoReN 编辑(叠进同一 codebook)→ chat 路径检索答对** 这条
完整接缝**真权重跑一次**,证明两半(consolidate 逻辑 / HoReN 机制)合在一起也成立。**不改任何被冻结的签名**。

## 为什么能全真、确定性、无网络(已实测)
- `dedup.classify`:registry 空 → 直接 `Decision("new")`(不调 LLM);之后 fact 彼此语义独立时,embedding 预筛
  shortlist 为空 → 仍 "new"(不调 LLM)。**唯一硬依赖 `embed.encode`**(本地 `all-MiniLM-L6-v2`,已验证离线可用,dim 384)。
- 选 eval/samples.json **前 4 条 type-A**:两两 cosine ≤ 0.60 < 0.85(已实测)→ 4 条全判 "new",全程不触 LLM/网络。
- `run_pass` 循环前一次性抓 `model = _model_provider()`(fresh 时是 raw HF);各 item 共享同一 HF 对象,
  `editing.edit` 把 native + chat key 叠进同一 adapter(正是 v0.5 unwrap 修复保证的 sequential 行为)。

## 实现(driver:`spikes/spike0_consolidate_seam_e2e.py`)
1. `model_host.load_base()` 一次;`store.reset()` 清空;`consolidate.set_model_provider(lambda: model_host.current_model())`。
2. 构 4 条 edit-route `MemoryItem`(route="edit", status="buffer", `provenance[PROV_EDIT]={stem,target,subject}`),
   `buffer.append` 进 buffer。**不经 extract**(extract 的 LLM 拆解另有单测;本 driver 验"从 buffer 往后"的接缝)。
3. `n = consolidate.run_pass("manual")`(**一次** pass,不 reload)。
4. 断言:`n == 4`(全 new 全写)、buffer 抽干(`load_unconsolidated()==[]`)、store 里 4 条 consolidated edit-route、
   每条 provenance 记了 `PROV_EDIT_REF`+`PROV_CONSOLIDATED_AT`。
5. 断言 codebook 单调叠进一个 adapter:`len(edit_module().keys) == 1 + 2·4 = 9`。
6. chat 路径(**空 RAG 窗口**)逐条问 stem → `score ≥ 0.85` 且答案含 target_new。
7. 无关 query → `score < 0.85`、无 target 泄漏。

## 绿灯判据

- 4 条 edit 叠进**一个** codebook(size==9)。
- 4 条从 chat 路径检索命中且答对;无关 query 不误触发。
