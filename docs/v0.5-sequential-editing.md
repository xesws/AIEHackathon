# v0.5 — Sequential editing:多条 fact 叠进同一个 codebook(修双层包裹 bug)

> 阶段:HOREN 后端纵切 / 串行 / GPU 常驻。开发前写计划,开发后补结果(遵 CLAUDE.md /docs 规则)。
> 这是 consolidate(buffer→weights 批量写)的**前置 blocker**:一次 pass 要顺序写 N 条 fact 进**同一个** codebook。

## 目标
让 `editing.edit` 可以在**同一个常驻模型**上被连续调用 N 次,N 条 edit **叠进同一个 HopfieldAdapter codebook**
(而不是把 wrapper 一层层套娃),并且每条都能从 **chat 推理路径(Plan B)** 检索命中。这是 Engram 核心动作
(buffer → 一次 consolidation pass → 多条 fact 写进一个 codebook → 全部可从 chat 检索)的底层保证。

## 根因(精读 third_party/horen 源码确认)
- `editing.edit` 把 `apply_horen_to_model` 返回的 **HOREN wrapper** 经 `register_edit_module` 存成常驻模型
  (`model_host._S["model"] = wrapper`)。
- 第 **2** 次 `editing.edit(model_host.current_model(), ...)` 于是把**wrapper**(而非底层 HF model)传进
  `apply_horen_to_model` → `HOREN(model=wrapper)`。
- `HOREN.__init__` 执行 `parent_module(wrapper, "model.layers.29.mlp.down_proj")`,沿 `wrapper.model.layers…` 走;
  但 `wrapper.model` 是 `LlamaForCausalLM`,它的子模块是 `.model` 不是 `.layers` → 报错 **"model.layers not found"**。
- 这就是"双层包裹"bug:第 2 条 edit 直接崩,根本写不进 codebook。

## 修法(报告已 scope,~2 行)
在 `editing.edit` 里:若传入的 `model` **已经是** `HOREN` wrapper,先**解包**到 `model.model`(底层 HF model)再
(重新)应用。这样 `HOREN.__init__` 会发现 `down_proj` 上**已装着** `HopfieldAdapter`(editor.py:51 的 `else`
分支 → `activate_params()`,**不**替换 adapter、不重置 codebook),`add_key` 把新 key/value **append** 进**同一个**
codebook → N 条 edit 顺序叠加。这正是 HoReN 原生 sequential editing 的契约(我们之前把 wrapper 存回去破坏了它)。
- **不重新设计 wrapper**;`key_mode='chat'`(Plan B 默认)路径不动;raw 路径不动;0.85 阈值、key 向量空间、
  query-span 逻辑全不动。

## 不做(明确 out of scope,用户已确认 buffer/consolidate 未建好)
- **不实现 `consolidate.run_pass`**:它现在是 `NotImplementedError` stub,依赖未建的 buffer/dedup/rag_store。
  `editing.edit` 已默认 `key_mode='chat'`,等 consolidate 真正建起来时**自动继承** chat keying,届时无需再改。
- 不加 robustness / retain-test / paraphrase 容忍(multi-key/judge)/ 非空 RAG 窗口写 key —— 都不在本轮 demo 内。

## 验证(手动 dry run,不依赖 buffer/consolidate —— 只证 HOREN pipeline 本身对)
新建 `spikes/spike0_sequential_edit.py`:
1. `load_base()` **一次**;取 eval/samples.json 前 **4** 条 type-A fact(已在 v0.3 e2e 单条验证过)。
2. 在**同一个常驻模型**上**顺序** `editing.edit(..., key_mode='chat')` 4 次 —— 断言:无双层包裹报错、
   codebook **单调增长**(1 placeholder + 每条 2 个 key[native raw + 追加的 chat] = 期望 1+2·4=9)。
3. 4 条全部 edit 完后,chat 路径(**空 RAG 窗口**)逐条提问 stem → 每条 `score ≥ 0.85`(命中)**且**答案含
   `target_new`。
4. 一条无关 query → `score < 0.85`(不误触发)、答案不含任何 target。
**N=4 即本验证的规模上限(= 任务 spec 的 3–4 条),sanity 出结果即停下报告,不自动冲全量。**

## 绿灯判据
- 4 次顺序 edit 在一个常驻模型上跑通,**无** "model.layers not found",codebook 单调增长到一个 adapter 里。
- 4 条 fact 各自从 chat 路径检索命中(≥0.85)且生成含 target_new。
- 无关 query 不误触发(<0.85、答案无 target)。

## 结果(v0.5 — PASS ✅,N=4 sanity)

`python spikes/spike0_sequential_edit.py` —— `load_base()` 一次,4 条 type-A fact 在**同一个常驻模型**
上**顺序** edit,全程**无** "model.layers not found"、**无**双层包裹。

| 检查 | 结果 |
|---|---|
| Codebook 单调叠加(单个 adapter)| ✅ size 3→5→7→9,= 期望 `1+2·4=9`;live adapter 即被叠的那一个 |
| 4 条 fact chat 路径(空 RAG)检索 | ✅ score `1.000 / 1.000 / 1.000 / 0.996`(全 ≥0.85 fire),答案全含 `target_new` |
| Locality(无关 query)| ✅ score `0.812 < 0.85`(不触发)、无 target 泄漏,答案正常("100 degrees Celsius") |
| 单条 edit 延迟 | 3.85–4.61 s/条(与 v0.2 单条 ~4.5s 一致,叠加未变慢)|

- **根因修复 = `editing.py` 解包 1 行**:`hf_model = model.model if isinstance(model, HOREN) else model`。
  第 2+ 次 edit 走 HF model,`HOREN.__init__` 命中已装的 `HopfieldAdapter`(`activate_params()`),
  `add_key` 把新 key/value append 进同一 codebook。wrapper / 阈值 / key 空间 / query-span 全未动。
- **未触碰 `consolidate.run_pass`**(仍是 stub,依赖未建的 buffer/dedup);`editing.edit` 默认 `key_mode='chat'`,
  consolidate 真正建起来时自动继承 chat keying。
- 备注:无关 query 0.812 离阈值 0.85 余量约 0.04(偏小但稳过);部分答案有 target 词重复
  (HoReN value 注入在贪心下推同一 token),不影响"含 target_new"判定。torch 全程 2.8.0+cu128。
