# v1.6 — contrast-out (codebook mean-centering) 修复 chat 路径 locality(measurement-first)

> 依据: v1.5 阶梯定位结论(`docs/v1.5-locality-debug-ladder.md`)+ 本轮用户指令
> 类型: **measurement-first** · contrast 实现成 toggle、对照 S3 测量、**本轮不上线**(flag default OFF)
> 目标: 验证「减掉 codebook 共享分量能重新撑开 intra-cone 距离 → 修复 locality(NEGxfire↓)而**不杀** PARA」这个**假设**(是待证伪假设,不是已知修复)。

## 背景锚点(别再质疑)
- v1.5 定位:chat keying(scaffold ⊕ flat-mean 池化)把整锥**抬过**固定 0.85 → siblings 误触发(S3 fact NEGxfire **11/20**、belief **7/20**)。JQ 框架被证伪(S0→S1 零变化)。见 [[horen-key-cone-collapse]]。
- 锥的病 = 所有 key 共享一个大分量(scaffold + question-stem + 池化 DC),cosine 被这个共享分量主导 → DIRECT≈NEG(整锥贴在 ~0.90)。
- 假设(B,待证伪):减掉这个共享分量后,残差能区分 fact-之间(NEG↓),但仍保留 fact-自身 src↔edit 的对齐(DIRECT/PARA 不掉)。

## 实现(additive,toggle,本轮只在 spike 内、不碰生产代码)
- contrast-out = 对每个 key 减去共享分量 `c`,再 **renormalize**,然后过**同一个**生产 deferral gate(`keying.score`→`adapter._query`,**阈值 0.85 不变**)。
  - `_query` 的最终分 `qt @ K.t()` 把 `K=self.keys` 原样点积、且不在入口归一化 q → key 必须是单位向量,故减 `c` 后**必须 renormalize**(read/write 都做)。
- 共享分量 `c`(读代码后择优,见 §选型)= **codebook 均值**(option a):`c = mean(所有 write-key)`。
  - 选 a 不选 b(empty-query scaffold baseline)的理由:a 参数-free、且天然把 scaffold + question-stem + 池化 DC **一并**包含(b 只估 scaffold 那一片);a 是 b 的**上位估计**。ROLE title 也明指 "codebook mean-centering"。
  - 两种粒度都报:**global**(全 10 条 write-key 的均值,= live 混合 codebook,主)+ **per-group**(fact / belief 各自均值,诊断锥间偏移是否要分桶)。
- ★ 关键:**同一个 `c` 同时减在 write-key 和 read-key 上**,再各自 renormalize,否则比较无意义。
- 本轮**不**改生产 keying/editor/serving;contrast 只活在 v1.6 spike 里(= flag default OFF、live 行为零改动 by construction)。数好看才由用户定下一步上线。

## 实验(复用 v1.5 harness、同 10 条真样本、同 seed、同 S3 keyer)
对 S3 配置(= live chat key)跑三条件,一张表:

| cond | 说明 |
|---|---|
| **S3** | 无 contrast(baseline,期望复现 v1.5:fact 11/20、belief 7/20)← 同时作完整性校验 |
| **S3+contrast(global)** | 减 global codebook 均值(全 10 write-key) |
| **S3+contrast(group)** | 减 per-group 均值(诊断) |

每条件、每组(fact/belief):per-sample DIRECT/PARA/最坏 NEG cosine + 汇总 `DIRECT 均值 | PARA 均值 | NEG 地板 | PARAfire(/5) | ★NEGxfire(/20)`。

## 成功判据(★联合判定,别误读)
真修复需**同时**:① NEGxfire 大幅↓(趋向 S0 的 0/20)= locality 回来;② PARAfire 保持高(≈5/5)= 编辑对自己的改写仍触发。
- 若 NEGxfire↓ 但 PARAfire **也**↓ → **false-fix 陷阱**(锥整体被平移到阈值下,跟 last_60/80 池化一个坑)→ 判 **FAILURE**。
- 判据是 (NEGxfire, PARAfire) 的**联合移动**,不是绝对分、不是单看 margin。两个数都显式报,并指明落在哪种情形(SUCCESS / FALSE-FIX / NO-EFFECT)。

## 也要报

- 若有效:幅度多少、PARAfire 是否完全保住。

## 结果(开发后)

### 落地 & 校验
- spike: `spikes/spike_v16_contrast_out.py`(measurement-only,独立 model 副本;contrast **只活在 spike 内** = live 代码零改动 / flag default OFF by construction;**未碰** server/codebook/memory/editing/samples.json)。
- **生产路径零漂移**:S3 key = `key_chat_span` == 生产 `compute_key(templated=True)`(drift-guard `allclose atol 1e-3` 通过);打分用 `keying.score`(生产 `_query` gate,β20/α0.1/iter1,**阈值 0.85 不变**)。
- **完整性闸**:S3 无 contrast baseline **精确复现 v1.5**(fact NEGxfire 11/20、belief 7/20、PARAfire 5/5)→ contrast 是 vs S3 的**唯一**新变量。
- contrast `c` = **codebook 均值**(单位 write-key 的质心):global(全 10)+ per-group。`||c_global||=0.901`、`||c_fact||=0.925`、`||c_beli||=0.923`,`cos(c_fact,c_beli)=0.905`。
- `torch 2.8.0+cu128 / True` 首尾一致;VRAM 16.2G;exit 0,无 traceback。

| cond | group | DIRECT | PARA | NEG地板 | NEGmean | PARAfire | **NEGxfire** |
|---|---|---|---|---|---|---|---|
| **S3**(无 contrast) | fact | 0.915 | 0.902 | 0.898 | 0.842 | 5/5 | **11/20** |
|  | belief | 0.925 | 0.932 | 0.871 | 0.827 | 5/5 | **7/20** |

### ★ 联合判定:FALSE-FIX(FAILURE)
四个格子全部 **FALSE-FIX**:NEGxfire 11/7 → **0/20**(locality 表面"修好"),但 PARAfire 5/5 → **0/5**、且 DIRECT 对**每一条**都掉到 0.85 以下 —— **整锥被平移/塌缩到阈值下,什么都不再触发**。这正是 ROLE 警告的 false-fix 陷阱(与 last_60/80 池化同坑),按判据判 **FAILURE**,不是成功。

### 为什么失败(机制)
- `||c_global||=0.901` → 共享分量是每个 key 的 **~90%**;锥太紧,质心几乎"就是" key 本身。减掉后残差 `k−c` 很小、被噪声主导,fact 身份没有以干净的正交残差形式留存。
- 残差里 DIRECT/PARA 绝对值塌到 0.40–0.66,远低于固定 0.85 → 固定阈值全拒(整锥下移)。
- **更关键:残差也救不回 facts。** 重标阈值能不能救?看残差可分性(per-sample,global):
  - **belief**:PARA{0.55,0.73,0.70,0.58,0.75} vs worstNEG{0.31,0.24,0.20,0.18,0.11} → **有 gap**(min-PARA 0.55 > max-NEG 0.31),重标阈值**可分**。
  - **fact**:PARA{0.33,0.36,0.44,0.60,0.58} vs worstNEG{0.15,0.35,0.24,0.11,0.16} → **重叠**(min-PARA 0.331 < max-NEG 0.349)→ **没有单一阈值能全开 PARA 而不开某条 NEG**。
- 即:contrast-out 没给 facts 留下可分结构;facts(本就更糟的组)恰恰是 contrast 也救不了的组。per-group 减得更狠(残差更小),DIRECT/PARA 更低 → 更糟。

### 也答 ROLE 的问
- **fact / belief 帮助一样吗?** 都 false-fix(都 0 PARAfire / 0 NEGxfire)。但残差结构不同:**belief 残差可被重标阈值救,fact 残差重叠、救不回**。
- **用哪种 c 估计?** codebook-mean,global(主,= live 混合 codebook)+ per-group(诊断);两者都失败,per-group 更糟。
- **有效幅度 / PARA 保住没?** 无效;PARA 完全没保住(5/5→0/5)。

### 更深的发现(跨 v1.5 + v1.6)
- 回看 **S3 baseline 本身**:fact A0002 的 sibling-NEG(0.898)就**高于它自己的 PARA**(0.886)→ **在这个 layer-29 flat-mean chat key 里,一条 JQ-fact 与另一条根本不线性可分**。既不能靠阈值分(facts 在 S3 就重叠)、也不能靠减均值分(残差仍重叠)。fact-身份信号埋在 scaffold+question-stem 噪声地板之下。
- 推论(**待用户定,不自动执行**):修复多半**不在对现有 key 做算术**(mean-center / 单纯重阈值都救不了 facts),而要换 **key 表示**本身。
