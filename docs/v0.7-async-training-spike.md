## 1. 问题陈述

### 1.1 现状:编辑是同步的、且在 serving model 上原地训练

当前 hero loop 的固化路径是同步阻塞的:
`consolidate.run_pass → editing.edit → apply_horen_to_model(model_host.current_model(), …)`
(`editing.py:62`)。`apply_horen_to_model` 跑 HoReN 的 forward+backward+optimizer
(`n_iter=50`,SPIKE 0 实测 ~4.5s,峰值 ~16.2G),而且**直接跑在常驻 serving model 的
`model.layers[29].mlp.down_proj` 槽位上**。两个后果:

1. **逻辑阻塞**:那 4.5s 里 chat handler 在等编辑跑完,这一轮对话被卡住。
2. **正确性污染(更严重)**:那 4.5s 里 live `down_proj` 是一个**半训练好的 adapter**
   —— 任何并发推理读到的不是"旧值"也不是"新值",而是**优化中途的 garbage 中间态**
   (`editor.py:86` 把 `training=True`、`editor.py:108` 每步 in-place 改 `values`)。

### 1.2 reframe:"异步"到底解决了什么(摘自 design doc)

把 editing 包成 async **只解决逻辑阻塞**(事件循环不卡),**完全不解决物理争用**
—— 一块 GPU 上训练的 fwd+bwd+optimizer 和推理的 fwd 抢同一批 SM、同一块显存、同一条带宽,
Python 层 async/线程让 kernel "并发提交",但 CUDA 在同一 stream 上仍串行执行 kernel。

> 真正让推理"不被打断"的不是 async 本身,而是另外两件事:
> (1) **shadow editor**:绝不在 serving model 上原地训练;
> (2) **物理隔离/限流**:把训练的 GPU 压力和推理在时间/空间上错开。
> async 只是这两件事的编排外壳。

`online_editing_design.md` 已经把这套理论写清楚了(难点 A/B/C/D),但**没有在我们这张
A6000 上实测过**:shadow 训练时 serving 真的能保持正确 + 还可用吗?atomic swap 真的不撕裂吗?
两份 model + 一次 edit 真的塞得进 48G 吗?**v0.7 = 把理论跑成一个可测量的 demo。**

## 2. 设计空间(难点 A/B/C/D,蒸馏自 design doc)

| 难点 | 一句话 | 本 spike 怎么处理 |
|---|---|---|
| **A 物理争用** | 单卡上训练 backward 抢 SM/带宽,推理 TTFT 必被顶高,**只能 bound 不能消除** | phase [3] **实测** serve-during-train vs idle 的延迟差,诚实报告 |
| **B swap 正确性** | **绝不原地训 serving model**;在独立 adapter 实例上训,训完一次性 atomic swap | **核心**:copy=True 影子训练 + `swap_edit_module` 原子提升(见 §3) |
| **C read-your-writes 一致性** | 异步 gap 里 fact 已"请求学习"但没进权重 → buffer 整段注入兜底;**先 swap 再 drop** | phase [6](N≥2)演示 swap-then-drop 顺序;buffer 兜底沿用 two-speed 设计 |
| **D 长期在线** | single-writer 串行队列 / canary / codebook 回收 / 持久化 / crash 隔离 | phase [4] 单写队列 + phase [5] crash 隔离;持久化与回收留 v0.8 |

## 3. 异步训练架构(本 spike 验证的形态)

### 3.1 三个被代码事实"锁死"的设计决策

落地形态不是随便选的,是被三条已验证的代码事实逼出来的:

- **一个 model = 一个 down_proj 槽 ⇒ 不可能在同一个 model 上同时 serve + train。**
  `model_host._S` 只持有一个常驻 model + 一个 `parent/attr` 槽(`model_host.py:24-32,69-76`)。
  要让 serving 在 4.5s 编辑期间保持正确,**训练必须发生在另一个 model 实例上**。
- **serving 必须走 HoReN wrapper。** `generate._greedy` 调 `model.generate`,而只有
  `HOREN.generate` 会在每次调用时 `setattr(eval(f"self.model.{self.layer}"), "key_id", …)`
  设置注入位置(`editor.py:67-73`)。关键:它用 `eval` **每次重新从槽位读 adapter**,
  **不持有旧引用** —— 所以**换掉槽里的 adapter 对象后,wrapper 的 key_id 会落到新 adapter 上**。
  这正是 swap-based 提升能成立的前提。
- **提升 = 原子换掉 live 槽里的 adapter 对象。** `swap_edit_module(adapter)` 是单次 `setattr`
  (`model_host.py:69-76`),GIL 下原子 —— 一次 forward 要么看到旧、要么看到新,不撕裂,
  旧 adapter 被在途 forward 的 refcount 兜住自然 drain。**绝不**用 in-place 改 live adapter
  的 `keys/values`(那是 3 次非原子 `torch.cat`,即 `_append_chat_key` 干的事,build 时安全、
  live 时会撕裂)。

### 3.2 影子怎么来:`copy=True`(deepcopy-on-edit)

`apply_horen_to_model(model, …, copy=True)` 会 `deepcopy(model)` 再在副本上训练
(`horen_main.py:22-23`),**传入的 live model 一根头发都不动**,且**无 `_S` 副作用**。
于是影子训练一行就够:

```python
wrapper, _ = apply_horen_to_model(live_hf, tok, [req], hp, copy=True)   # 训练发生在 deepcopy 上
new_adapter = <wrapper 槽里的 adapter>                                   # 训好的 codebook
```

- **carryover 自动**:deepcopy 复制的是 live model 的**当前**状态(含当前 codebook),所以第 k 次
  编辑 deepcopy 出来的影子天然带着前 k-1 条编辑 → `HOREN.__init__` 见到已存在 adapter →
  `activate_params` + `add_key` 追加第 k 条(`editor.py:51-56, 263-280`)。无需手动搬 codebook。
- **代价**:每次编辑 deepcopy 一份 8B(瞬时 +16G,峰值 ~32G;编辑结束副本释放)。这是 spike
  为"简单 + 正确隔离"付的显存税。**省显存的 frozen-base aliasing(S-LoRA 式,只复制小 adapter)
  是 design doc 推荐的生产路径,留 v0.8**(§8)。

### 3.3 提升怎么做:transplant + atomic swap(请求边界)

训好的 `new_adapter.layer` 指向**副本**的 down_proj Linear(权重与 live 完全相同,因同源预训练)。
提升时:把 `new_adapter.layer` 重指回 `_S["original"]`(live 的冻结 Linear)解耦副本,再
`swap_edit_module(new_adapter)` 原子换进 live 槽。之后副本 model(16G)无人引用即释放,只留下小
codebook + 共享的冻结 Linear。

**请求边界原子性靠一把 `swap_lock`**:serving 线程在每次 `generate` 全程持锁(短,~几十 token),
worker 只在**那一瞬间的 swap** 抢锁 —— **训练的 4.5s 全程不持锁**,所以推理与训练照常并发
(GPU 争用,实测),但 swap 永远发生在两次请求之间,绝不在一句话的 token N 与 N+1 之间。

### 3.4 数据流图(对齐 `engram_async_editing_serving.mermaid`)

```
 SERVING 线程(reader,forward-only,永不被打断)      EDITING worker(1 线程 = 单写者)
   with swap_lock:                                       queue.Queue() 装 pre-split edit 请求
     generate(model=current_model(), chat)               每条:
   读 live 槽里的 LIVE adapter                              1. apply_horen_to_model(copy=True)
        │  全程正确 + 在线                                     → deepcopy 上训练 + append chat key
        ▼                                                   2. new_adapter.layer ← _S["original"]
   ───────── 一张 A6000,SM/HBM 共享 ─────────              3. with swap_lock:  ① ATOMIC
        ▲   (难点 A:争用只能 bound)                            swap_edit_module(new_adapter)
        │                                                   4. buffer.drop(② 必须在 swap 之后)
   submit() 非阻塞入队 ───────────────────────────────►
```

## 4. Spike 实验设计(`spike_v07_async_training.py`)

SPIKE-0 风格:module docstring + `sys.path` 注入 + `load_type_a` + 编号 banner 分阶段 +
`PASS ✅ / CHECK ⚠️` verdict。样本取 `eval/samples.json` 前若干 `sample_type=="A"`
(`stem=edit_prompt`,`paraphrase=queries[0].q`,`target=target_new`)。复用:
`apply_horen_to_model` / `HOREN` / `HopfieldAdapter` / `swap_edit_module` / `edit_module` /
`_S["original"]` / `keying.compute_key` / `generate.generate`。

| phase | 证明什么(对应难点) | 指标 / 判据 |
|---|---|---|
| **[1] bring-up** | 两份 model + edit 塞得进 48G | `HOREN(hp, live_hf)` 装空 codebook→`_S["model"]`;load 后 + edit 期 **峰值 GPU**。GATE:< ~40G |
| **[2] 非阻塞 accept** | 逻辑 async | `submit(fact0)` 入队即返回 **<50ms**,而 worker `train_seconds≈4.x s` 在后台跑 |
| **[3] serve-during-train(头条,难点 B)** | 影子隔离:训练期 serving 仍正确 | 训练中对 live 问**无关**问题:答案**正常不变** + fact0 **尚不可知**;记 **during-train vs idle 延迟**(难点 A,诚实报数) |
| **[4] atomic 提升(难点 B swap)** | swap 前后语义翻转、不撕裂 | swap 前 `score(fact0)<0.85`/答非 target;swap 后 `score≥0.85` 且 chat `generate` 答出 `target_new`;locality 探针仍 `<0.85` |
| **[5] crash 隔离(难点 D.5)** | 影子训崩 → serving 无感 | worker 某条训练中途 raise → serving **不受影响**(live adapter 没碰过),队列继续 drain |
| **[6] 单写串行 + swap-then-drop(难点 D.1+C,N≥2,sanity 过后才跑)** | 串行队列 + carryover + 顺序 | 连提 2-3 条:逐条处理、codebook 单调增长、各自 swap 后可检索、locality 守住;`buffer.drop` 在 swap **之后** |

**护栏:sanity = phase [1]-[5] @ N=1,出结果后停下报告,等确认再跑 phase [6]/N≥2。**

## 8. 局限与遗留(→ v0.8)

- **frozen-base weight aliasing**:本 spike 用 `copy=True` 每条编辑 deepcopy 整份 8B(瞬时 2×base)。
  design doc 推荐的生产形态是 serve/edit 两实例**共享同一份冻结 base 权重**(只复制小 adapter +
  其 activation,16G base 不翻倍,multi-LoRA serving 的 pattern)。这是下一步的核心优化。
- **物理争用是本质矛盾(难点 A)**:单卡上 serve-during-train 的延迟抬升消不掉,只能靠"edit 短、
  不频繁、可延后/限流"去 bound。双卡物理隔离才是干净解,但 2× 成本。
- **持久化(难点 D.4)**:adapter 只在内存里,进程重启全丢。需把 codebook 序列化落盘 / 从 edit log
  replay。`model_host` 目前没持久化 adapter。
- **codebook 无界增长(难点 D.3)**:supersede/retire 要真的从 codebook 回收,否则单调增长拖慢检索。
- **canary(难点 D.2)**:每次 edit 后跑"刚编 fact + 固定 locality 探针"的 cheap 在线 canary,
  掉了就告警/回滚;本 spike 只做了 phase [4] 的雏形。
- **多租户天花板(难点 C)**:权重 edit 是全局状态,weight-level personalization 天生没法 per-user
  隔离(buffer/RAG 能,weights 不能)。
- **真正接进 `consolidate.run_pass`**:本 spike 是独立 harness;把这套 queue+shadow+swap 接进
  consolidation 主路径(并保持现有模块零改动 / 最小改动)是 v0.8 的落地工作。
