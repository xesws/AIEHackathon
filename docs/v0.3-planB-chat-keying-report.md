## 1. Problem Statement

### 1.1 现象
SPIKE 0(v0.2)证明 **RAW-prompt 路径**绿:`edit → retrieve → answer` 成立(token-EM=1.0)。
但 Engram 真正的推理入口是 **CHAT 路径**(`build_prompt` → `apply_chat_template`)。编辑一个 fact 后,
用 chat 模板提问,**编辑不触发**,模型回到 base 行为:

> edit:`"What university did Watts Humphrey attend?" → "Illinois Institute of Technology"`
> chat 提问 → base 答:`"I'm not aware of any information about a person named Watts Humphrey…"`

这是 hero loop(在线编辑 → chat 对话里立刻生效)的 **#1 阻塞**(见 `docs/memory/DESIGN.md` §10、`docs/v0.2-spike0.md` 头号发现)。

### 1.2 根因(code-level)
HoReN 是 external-memory 编辑法:`HopfieldAdapter` 替换 `model.layers[29].mlp.down_proj`,
检索 key = 输入前向在 **layer-29 的隐状态**:

```
key = normalize( _select_query(layer29_input, key_id) )          # 策略 last_60_perc_prompt_tokens_avg
score = _query(key) ≈ cosine(key, codebook_keys)                 # 归一化 Hopfield, max_iter=1
触发 ⟺ max(score) > 0.85
```

write 与 read 落在**不同输入分布**上:

| | 输入前向跑在 | `key_id` | key 实际覆盖 |
|---|---|---|---|
| **WRITE**(`HOREN.edit`) | RAW `"{stem} {target}"` | 最后一个 raw-stem token | 末 60% 的 **raw stem** token |
| **READ**(`HOREN.generate`) | **CHAT 模板化**序列 | `input_ids.shape[1]-1`(末 token) | 末 60% 的 **整条模板序列** |

CHAT 序列 = `<bos> system + RAG窗口 + role headers + user-turn + assistant header`。于是 read-key 的
末 60% 平均窗口里 **绝大部分是固定 scaffold**(system+窗口+headers),user-turn 只占一小段 → read-key 偏离
raw write-key → `score < 0.85` → 门控关闭 → value 不注入 → 编辑不触发。

### 1.3 为什么不是简单挪一下位置(Plan A 被放弃)
最初考虑过两种 "Plan A":(a) read-key 仍用末 token、(b) read-key 用 user-turn 最后一个 content token。
两者都仍走 `last_60_perc_avg`,即 **把 [0..key_id] 末 60% 平均**,这段对 chat 序列而言 **~86% 是 scaffold**
(实测:hero query 的 60% 窗口里 query token 仅占约 8/58)。后果:
1. 不同 fact 的 key 都被同一份 scaffold 拉到一起 → **key 坍缩 → locality 崩**(无关 query 也会 >0.85 误触发);
2. content-token 还会把 value **注入位置**前移到答案槽之前,损害值迁移。

结论:问题的本质不是"挪 key_id",而是 **key 里混进了前缀**。必须从 key 里**剔除 scaffold 行**。

## 2. Plan B 修复逻辑

### 2.1 核心 insight —— 在**隐状态张量的行**上拆 query(不是拆文本)
整条 templated prompt 跑一次 forward,在 layer-29 得到 `[seq_len, hidden_dim]` 的隐状态;做 key 时
**只取 user-turn(query)token 对应的那几行**做 mean-pool,排除 scaffold 行。write 与 read 用**同一套切片规则**。

```
  layer-29 隐状态(整条 templated prompt):
  ┌──────────────────────────────────────────────────────────────────┐
  │ [sys 行] [RAG窗口 行] [role-header 行] [QUERY-span 行] [assistant header 行] │
  └──────────────────────────────────────────────────────────────────┘
                                         └── key = mean(这几行) ──┘
```

为什么对:scaffold 行不进 key →
- write(stem)与 read(同 stem)render 完全相同 → 同样的 query 行 → key 相同 → score≈1.0(① 解决);
- write(stem)与 read(无关 query)→ query 行内容不同 → 不再因共享 scaffold 而坍缩(locality 保住);
- 对**非空 RAG 窗口**天然鲁棒:窗口行无论装什么都被排除在 key 外(本轮只验证空窗口 hero)。

### 2.2 关键设计决策(及理由)

**决策 1 — 注入位置不变(末 token),只改 key 计算。**
`forward` 里 `key_id`/`last_prompt_token_index` 同时承担两件事:① key 池化窗口的末端、② value 注入位置。
Plan B **解耦**这两者:key 改用 query-span 行池化,但 **注入仍在末 token**(`input_ids.shape[1]-1`)。
原因:value 是在 raw `"stem target"` 上训练的——它学到的是"在'下一 token 应为 target'的位置加上 V 能让 logits 预测
target[0]"。chat 推理时,"下一 token = 首个生成 token"的位置正是**末 prompt token**(assistant header 之后)。
所以注入末 token = 复刻 raw 训练几何 → value 迁移最佳。(content-token 注入会前移到答案槽前,更差。)

**决策 2 — write key 用 APPEND 不 overwrite。**
`editing.edit(key_mode='chat')` 保留 HoReN 原生 raw key,**追加**一个 Plan-B chat key,**复用同一个已训练的 value 行**。
一次编辑后 codebook 同时有 raw key 与 chat key:raw 提问命中 raw key、chat 提问命中 chat key,二者注入同一 value
(`_query` 取 max,只注入最佳那个,无双重注入)。这样 **raw 路径零回归**,SPIKE 0 全绿。
注:这是**同一 stem 的两种 render**,不是 paraphrase multi-key(不碰 ②);`key_mode='raw'` 退回 legacy 单 raw key。

**决策 3 — span 定位在**真实 render** 上,用 offset_mapping。**
`query_span_in_rendered(tok, rendered, text)`:对**实际喂给模型的** render 串做 `tok(..., return_offsets_mapping=True)`,
用 `rendered.rindex(text)` 取 user-turn 字符区间,保留**与该区间重叠且非空 offset** 的 token(special/role token 的
offset 为空,自动排除;重叠判定对 BPE 边界合并与 double-BOS 鲁棒)。read 侧定位在真实 prompt_text 上,所以
**未来非空窗口也能正确定位 query 行**(只是 write 仍是空窗口 render,变量窗口下 write/read 上下文不一致会压低分,留作下一步)。

**决策 4 — 提取逻辑单一来源,write/read 不分叉。**
新增 `HopfieldAdapter._pool_span(layer_input, start, end)`(对 [start,end] 行 mean),**被 `forward`(read)和
`keying.compute_key`(write)共用**。`compute_key` 用一个 `forward_pre_hook` 抓 `args[0]`(= down_proj 输入,与是否注入无关),
临时 `adapter_mode='none'` 跑一次纯捕获前向,再调 `adapter._pool_span` / `_select_query` → 与 `forward` 用的是**同一份代码**。

### 2.3 数据流

```
WRITE  editing.edit(stem→target, key_mode='chat')
  1) HoReN 训练:RAW "stem target" 上 n_iter=50 → 学到 value V,存 raw key(HoReN 原生,不动)
  2) _append_chat_key:
       render = apply_chat_template(build_prompt(stem, [], []), add_generation_prompt=True)   # 空 RAG 窗口
       前向(mode='none', pre-hook 抓 layer29 输入)→ _pool_span(stem 的 query 行) → normalize → chat_key
       codebook += chat_key ; values += V(复制) ; key_labels += label(复制)

READ   generate(query, use_chat_template=True, with_rag=False)
  1) prompt_text = apply_chat_template(build_prompt(query, [], []), add_generation_prompt=True)
  2) adapter.query_span = query_span_in_rendered(tok, prompt_text, query)        # 定位 query 行
  3) model.generate → HOREN.generate 设 key_id=末token(注入位)→ HopfieldAdapter.forward:
       query = normalize(_pool_span(args[0], *query_span))      # 只取 query 行
       max(_query(query)) > 0.85 ? 命中 chat_key → 在末 token 注入 V → 生成 target
```

`stem==query` 时 write/read render 一致 → 同样 query 行 → score≈1.0。

### 2.4 改动清单
| 文件 | 改动 |
|---|---|
| `editor.py` | `HopfieldAdapter` 加 `self.query_span=None`、`_pool_span()`;`forward` 在 `query_span` 非空时用 `_pool_span` 取 key,否则 legacy。**阈值/`_query`/注入位/key 向量空间不动。** |
| `keying.py`(新) | `compute_key`(pre-hook 捕获 + 复用 `_pool_span`/`_select_query`)、`query_span_in_rendered`(offset-mapping 定位)、`score`(用 `_query` = 生产门控) |
| `editing.py` | `edit(key_mode='chat')` 编辑后 `_append_chat_key`(append chat key,复用 value) |
| `generate.py` | chat 分支设 `adapter.query_span`,raw 分支清 `None`(防跨调用污染) |

## 3. 测试结果与分析

分数均用 adapter 自身的 `_query`(= 生产 0.85 门控函数;诊断时 float32,bf16 门控相差 <~0.01)。

### 3.1 诊断 grid(`spike0_planB_keying_grid.py`,5 个 type-A)

codebook 先置成单个 RAW write-key,再对四种 read-key 打分(均值):

| | read raw | read chat (Plan B) |
|---|---|---|
| **stem** | **1.000**(对照,keying sane) | 0.892(① 单独成本) |
| **paraphrase** | 0.851(★ ② 裁决) | 0.820(①+②) |

**修复验证**(codebook 置成 CHAT write-key):

| | 均值 | 逐样本 | 解读 |
|---|---|---|---|
| chat/stem | **1.000** | 0.998–1.004 | 同 render → key 相同 → 完美命中 |
| chat/paraphrase | **0.913** | 0.900–0.924,**5/5 PASS** | ① 去除后,paraphrase 也过线,且 **高于** raw/para 0.851 |
| UNRELATED(off-topic) | **0.789** | 0.726–0.820,**0/5 触发** | < 0.85 → **locality 保住** |

**逐 cell 分析:**
- **raw/stem=1.000**:对照通过,证明整套 keying+打分 sane。
- **chat/stem=0.892(vs raw codebook)**:即使拿**旧 raw codebook** 去比,Plan-B 的 chat-stem key 也已 0.892>0.85
  —— 说明 span 隔离本身就把 chat key 拉到了 raw key 附近(scaffold 不再稀释);存了 chat key(修复)后升到 1.000。
- **raw/para=0.851(② 裁决)**:这是 paraphrase 在 **raw 空间**的真实容忍度,**卡在阈值上**(逐样本 A0004=0.845、
  A0005=0.833 略低,2/5 不过)。判定:**② 处于边界,不是干净的"非问题"**。
- **chat/para=0.913 > raw/para=0.851**:为什么 chat 反而更高?因为 write(stem)与 read(para)的 query 行都
  **透过同一份固定 scaffold**(system + 空窗口)计算——query token 在因果注意力下 attend 到**完全相同**的前缀,
  这层共享上下文对**相关** query 有轻微"平滑/拉近"作用。关键是它**不**同等抬升无关 query(UNRELATED 仅 0.789),
  para(0.913)与 unrel(0.789)之间有 **~0.12 的 gap** → 这是**适度泛化**,不是整体 inflation。

### 3.2 生成 e2e(`spike0_planB_e2e.py`,2 个 type-A,每 fact 新 load)

grid 只证明"检索分数过线";e2e 证明"注入的 value 真能让 chat 路径**生成** target":

| fact | chat/stem | chat/para | chat/unrel | raw/stem | 绿 |
|---|---|---|---|---|---|
| A0001 (nickel buckles) | 0.996,答 `'nickel buckles…'` | 0.910,答对 | 0.781 不触发,答 `'The boiling point of water…'` | 答对 | ✅ |
| A0002 (pickled quail eggs) | 1.000,答 `'pickled quail eggs.'` | 0.910,答对 | 0.820 不触发,答 `'…is 100'` | 答对 | ✅ |

**VERDICT:hero loop(chat 路径,RAG off)2/2 PASS ✅。**
- **① 修复证实**:chat 路径现在**检索+生成** target(此前是 "I'm not aware…")。
- **值迁移证实**(最大的"grid 看不见"的风险):raw 训练的 value 在 chat 末 token 注入后,确实生成 target(两 fact 都对)。
- **locality 证实**:unrel 不触发,且回答是**正常 off-topic 内容**,没有被 target 污染。
- **raw 路径完好**:raw/stem 仍答对(append 保住 raw key)。

### 3.3 SPIKE 0 回归(`spike0_edit_swap_generate.py`)

全绿,关键是 **step [6](曾失败的 chat 路径)现在 fires**:

## 4. 局限与遗留(→ v0.4)
1. **sequential editing 缺口**:`editing.edit` 第二次编辑会把 HOREN wrapper 再包一层
   (`apply_horen_to_model(wrapper)` → `parent_module` 找不到 `layers`)。本轮 e2e 用"每 fact 重载"绕开;
   Engram 累积编辑(核心场景)需让 `editing.edit` 在入参已是 HOREN wrapper 时 unwrap 到底层 HF model(1–2 行)。
2. **② 边界 + locality 余量中等**:raw/para 0.851 贴阈值;chat/para 0.913 vs unrel 0.789 gap ~0.12。
   UNRELATED 目前是单条 off-topic;**同 subject(JQ)的不同 fact** 是更难的 locality,需扩大集合复核(若收窄,
   才考虑 ② 的 C/D 方案——本轮明确不做)。
3. **非空 RAG 窗口**:本轮只接线/验证空窗口 hero。读侧 span 已在真实 render 定位可推广,但 write key 仍是空窗口
   render → 变量窗口下 write/read 上下文不一致会压低分,留作 query-span isolation 的下一步。
